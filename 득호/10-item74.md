# 메서드가 던지는 모든 예외를 문서화하라

메서드가 던지는 예외는 런타임 중 발생하는 오류를 알려주는 중요한 정보다.
따라서 메서드가 던지는 모든 예외를 문서화해야 한다. 자바에서는 `@throws` 태그를 사용하여 메서드가 던지는 예외를 문서화할 수 있다.

특히 검사 예외는 항상 따로따로 선언하고, 각 예외가 발생하는 상황을 자바독의 `@throws` 태그를 사용하여 정확히 문서화하자

- 공통 상위 클래스 하나로 뭉뚱그려 선언하는 일은 피하자
  - Ex) `Exception`이나 `Throwable` 같은 최상위 클래스는 너무 포괄적이어서 메서드가 실제로 어떤 예외를 던지는지 알 수 없다.
    이는 같은 맥락에서 발생하는 여러 예외를 하나로 처리되어 정확히 문제를 진단하기 어렵다.
 

- 비검사 예외도 문서화하는 것이 좋다.
  - 일반적인 프로그래밍 오류를 뜻하는데, 자신이 일으킬 수 있는 오류들이 무엇인지 알린다면
    이를 사용하는 프로그래머는 자연스럽게 방어적으로 코딩할 수 있다.
  
  - 비검사 예외를 문서로 남기는 일은 특히 인터페이스 메서드에서 중요하다.
    - 문서화 내용을 기반으로 인터페이스의 구현체가 일관되게 동작하게 한다.
    
- 비검사 예외는 메서드 선언의 `throws` 목록에 넣지 않는 방식으로 구분하자.
  ```java
  /**
  * 파일을 읽어옵니다.
  *
  * @throws IOException 파일을 찾을 수 없거나 읽기 실패 시 발생 (검사 예외)
  * @throws IllegalArgumentException fileName이 null이거나 빈 문자열일 때 발생 (비검사 예외)
    */
    // throws 절에는 IOException(검사 예외)만 명시하여 구분한다.
    public void readFile(String fileName) throws IOException {
    if (fileName == null || fileName.isEmpty()) {
    throw new IllegalArgumentException("파일명은 필수입니다.");
    }
    // ... 파일 읽기 로직 ...
    }
    ```

- 한 클래스에 정의된 많은 메서드가 같은 이유로 동일한 예외를 던진다면 클래스 레벨어 설명을 추가하자.
  - Ex) `NullPointerException` 이 클래스의 모든 메서드는 인수로 null이 넘어오면 해당 예외를 던진다.

### 결론

메서드가 던질 가능성이 있는 모든 예외를 문서화 하는 것은 API 사용자에게 중요한 정보를 제공하여 올바르게 예외를 처리할 수 있도록 돕는다.
