# 리플렉션보다는 인터페이스를 사용하라

리플렉션 기능을 이용하면 프로그램에서 임의의 클래스에 접근할 수 있다.

이 개념을 처음 알았을 때 매우 좋은 기능이라고 생각했지만 돌아보면 
객체지향의 특징중 하나인 캡슐화라는 개념을 완전히 무시하는 기능이였다.

또한, 이를 공부하고자 했을때 문법이 복잡하고 어렵게 느껴졌다.

아무튼 리플렉션은 `Class` 객체가 주어지면

- `Constructor`
- `Method`
- `Filed`

를 사용해 연결된 실제 생성자, 메소드, 필드를 조작할 수 있다.

예를 들어 `Method.invoke`는 클래스가 가진 객체의 어떠한 메서드라도 호출할 수 있게 해준다.
리플렉션을 통해 컴파일 당시에 존재하지 않던 클래스도 이용할 수 있지만, 단점이 존재한다.

> 자바 9 이상부터는 모듈 시스템이 도입되면서 `setAccessible(true)` 메서드로 접근 제어를 무시하는 것이 불가능해졌다.
> 이로 인해 `InacessibleObjectException`이 발생할 수 있다.


### 컴파일타입 타입 검사가 주는 이점을 누릴 수 없다.
### 리플렉션을 이용하면 코드가 지저분하고 장황해진다.
### 성능이 떨어진다.
리플렉션을통산 메서드 호출은 일반 메서드 호출보다 훨씬 느리다.

코드 분석 도구나 의존관계 주입 프레임워크처럼 리플렉션을 써야하는 복잡한 경우가 있지만
이마저도 점차 줄이고 있다고 한다.

이를 위해서 리플렉션은 아주 제한된 형태로만 사용해야 단점을 피하고 이점만 취할 수 있다고한다.

### 리플렉션은 인스턴스 생성에만 쓰고, 이렇게 만든 인스턴스는 인터페이스나 상위 클래스로 참조해 사용하자

```java
public static void main(String[] args) {
        // 클래스 이름을 Class 객체로 변환
        Class<? extends Set<String>> cl = null;
        try {
            cl = (Class<? extends Set<String>>) // 비검사 형변환!
                    Class.forName(args[0]);
        } catch (ClassNotFoundException e) {
            fatalError("클래스를 찾을 수 없습니다.");
        }

        // 생성자를 얻는다.
        Constructor<? extends Set<String>> cons = null;
        try {
            cons = cl.getDeclaredConstructor();
        } catch (NoSuchMethodException e) {
            fatalError("매개변수 없는 생성자를 찾을 수 없습니다.");
        }

        // 생성자를 이용해 Set<String> 인스턴스를 반환할 수 있다.
        Set<String> s = null;
        try {
            s = cons.newInstance();
        } catch (IllegalAccessException e) {
            fatalError("생성자에 접근할 수 없습니다.");
        } catch (InstantiationException e) {
            fatalError("클래스를 인스턴스화할 수 없습니다.");
        } catch (InvocationTargetException e) {
            fatalError("생성자가 예외를 던졌습니다: " + e.getCause());
        } catch (ClassCastException e) {
            fatalError("Set을 구현하지 않은 클래스입니다.");
        }

        // 생성한 집합을 사용한다.
        s.addAll(Arrays.asList(args).subList(1, args.length));
        System.out.println(s);
    }

    private static void fatalError(String msg) {
        System.err.println(msg);
        System.exit(1);
    }
}
```
위 코드는 실행 시점에서는 Set 인터페이스를 활용하여
객체 생성에만 사용하고 생성된 객체는 Set 인터페이스로 참조하여 사용하는 좋은 예시이다.

### 그래도 단점
1. `try-catch` 블록이 많아져서 코드가 지저분해진다.
2. 컴파일 타임에 타입 검사가 불가능하다.(런타임으로 미루게 된다.)


### 결론
리플렉션은 복잡한 특수 시스템을 개발할 때 아니면 사용하지 말자
만약 사용하는 경우 인터페이스나 상위 클래스 참조를 사용해 객체 생성에만 사용하자
