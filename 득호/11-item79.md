# 79. 과도한 동기화는 피하라

> 핵심 원칙: 동기화 블록 안에서는 통제할 수 있는 코드만 실행하라
> 외부에서 온 코드(외계인 메서드)는 호출하지 마라

### 왜 위험한가?
과도한 동기화는 세 가지 문제를 일으킨다.

응답 불가(Deadlock): 서로 다른 스레드가 상대방의 Lock을 기다리며 영원히 멈춘다.
안전 실패(Safety Failure): 프로그램은 돌아가지만, 데이터가 훼손되거나 계산 결과가 틀어진다.
성능 저하: Lock 경합이 심해지면 처리량이 급격히 떨어진다

책에서는 동기화 블록 안에서 호출하는 메서드 중, 통제할 수 없는 메서드를 외계인 메서드라고 부른다.

### 외계인 메서드 유형
- 콜백 / 리스너
- 재정의 가능한 메서드
- 함수 객체
- 전략 패턴의 전략 객체

다형성을 활용하는 메서드들은 클라이언트가 어떤 구현을 넘겨줄지 알 수 없기 때문에 외계인 메서드가 될 가능성이 높다.

## 문제 코드와 해결 방법

**잘못된 코드 예시**

이후 나오는 모든 문제 상황은 아래 코드를 기반으로 한다.
```java
@FunctionalInterface
public interface SetObserver<E> {
    // ObservableSet에 원소가 더해지면 호출된다.
    void added(ObservableSet<E> set, E element);
}
```
`@FunctionalInterface`이므로 람다, 익명 클래스, 메서드 참조 등 어떤 구현이든 들어올 수 있다.
따라서 내부적으로 `added()` 메서드가 어떤 일을 할지, 얼마나 오래 걸릴지, 심지어 예외를 던질지도 알 수 없다.

### 1. ConcurrentModificationException

```java
public class ObservableSet<E> extends ForwardingSet<E> {
    private final List<SetObserver<E>> observers = new ArrayList<>();

    public void addObserver(SetObserver<E> observer) {
        synchronized (observers) {
            observers.add(observer);
        }
    }

    public boolean removeObserver(SetObserver<E> observer) {
        synchronized (observers) {
            return observers.remove(observer);
        }
    }

    // ❌ 동기화 블록 안에서 외계인 메서드를 호출한다
    private void notifyElementAdded(E element) {
        synchronized (observers) {
            for (SetObserver<E> observer : observers)
                observer.added(this, element);  // added()는 재정의 가능한 외계인 메서드!
        }
    }
}
```
어떻게 구현되어 있는지 모르는 `SetObserver`의 `added()` 메서드를 동기화 블록 안에서 호출하는 것을 볼 수 있다.

만약 내부에서 `added()` 메서드가 `removeObserver()`를 호출한다면, `notifyElementAdded()` 메서드는 `ConcurrentModificationException`을 던질 것이다.

```java
ObservableSet<Integer> set = new ObservableSet<>(new HashSet<>());

set.addObserver((s, e) -> System.out.println(e));  // 단순 출력

	set.addObserver((s, e) -> {
	System.out.println(e);
    if (e == 23) s.removeObserver(???);  // 동기화 블록 내부에서 removeObserver() 호출 
});
```

위 코드는 ObservableSet의 added() 메서드를 람다로 구현하고 있는데
내부적으로 `removeObserver()`를 호출하는데, 이 메서드는 `notifyElementAdded()`의 동기화 블록 안에서 호출된다.

이때 자바의 경우 동일한 스레드의 경우 Lock을 재진입 할 수 있어, 대기상태가 아닌 상태로 `removeObserver()`가 실행되지만,
결과적으로 순회 중인 리스트를 수정하게 되어 `ConcurrentModificationException`이 발생한다.

> 자바의 고유 락은 재진입을 허용하여 교착상태는 안 걸린다.
> 하지만 안전 실패(데이터 훼손) 문제가 발생하여 오히려 디버깅이 더 어려워진다.

### 사실 이부분이 이해가 안됐다.
**자바는 Lock 재진입 가능한데 동일한 스레드가 수정하는건데 왜 예외가 발생하지?**

1. ConcurrentModificationException의 진짜 의미
   이 예외는 사실 ModificationDuringIterationException으로 이해하는게 맞다.
   하지만 자바에서는 해당 네이밍을 가진 예외가 존재 하지 않고 ConcurrentModificationException이 그 역할을 한다고 한다.

### 2. 교착상태
```java
set.addObserver(new SetObserver<>() {
    public void added(ObservableSet<Integer> s, Integer e) {
        System.out.println(e);
        if (e == 23) {
            // 백그라운드 스레드에서 removeObserver를 시도
            ExecutorService exec = Executors.newSingleThreadExecutor();
            try {
                exec.submit(() -> s.removeObserver(this)).get();
                // 메인 스레드가 잡고이는 Lock을 백그라운드 스레드가 기다리는 상황이 발생한다.
            } catch (ExecutionException | InterruptedException ex) {
                throw new AssertionError(ex);
            } finally {
                exec.shutdown();
            }
        }
    }
});
```

백그라운드에서 실행되는 `removeObserver()` 메서드는 `notifyElementAdded()`의 동기화 블록이 잡고 있는 Lock을 기다리게 된다.
하지만 `notifyElementAdded()` 메서드는 `added()` 메서드가 끝나야 Lock을 풀기 때문에, 서로가 서로의 Lock을 기다리는 교착상태가 발생한다.

## 해결방법

### 1. 열린 호출 (opend calls)

동기화 블록 밖으로 외계인 메서드를 이동시키자

```java
private void notifyElementAdded(E element) {
    List<SetObserver<E>> snapshot;
    synchronized (observers) {
        snapshot = new ArrayList<>(observers);  // 복사
    }
    // 동기화 영역 바깥에서 snapshot을 가지고 외계인 메서드 호출
    for (SetObserver<E> observer : snapshot)
        observer.added(this, element);
}
```
>열린 호출(Open Call)이란?
>동기화 없이 외부 메서드를 호출하는 것을 말한다.
>Lock을 잡지 않은 상태에서 호출하므로, 외계인 메서드가 내부적으로 다른 Lock을 잡아도 교착상태가 발생하지 않는다.

### 2. CopyOnWriteArrayList 사용

자바에서 제공하는 스레드 안전한 자료구조를 사용하자

```java
private final List<SetObserver<E>> observers = new CopyOnWriteArrayList<>();

public void addObserver(SetObserver<E> observer) {
    observers.add(observer);  // 동기화 불필요
}

public boolean removeObserver(SetObserver<E> observer) {
    return observers.remove(observer);  // 동기화 불필요
}

private void notifyElementAdded(E element) {
    for (SetObserver<E> observer : observers)  // 스냅샷 순회
        observer.added(this, element);
}
```

CopyOnWriteArrayList의 동작 원리:

읽기(순회): 내부 배열의 스냅샷을 사용하므로 Lock이 필요 없다. 매우 빠르다.
쓰기(add/remove): 내부 배열 전체를 복사한 뒤 수정한다. 느리다.
적합한 경우: 읽기가 쓰기보다 압도적으로 많은 경우 (옵저버/리스너 패턴이 딱 이 경우다)

### 실무에서 자주 만나는 패턴들
- Spring의 `@EventListener`
    - 트랜잭션 커밋 전에 이벤트를 발행하는 경우 문제가 발생할 수 있다.
    - 이를 위해 @TransactionalEventListener(phase = AFTER_COMMIT)등의 옵션을 사용하여 안전하게 사용하자

- Lock 내부에서 Stream / Lambda 사용 조심하기
    - Stream이나, Lambda는 내부적으로 외부에서 온 메서드를 호출할 가능성이 높다.
    - 특히 Stream의 `forEach()`는 내부적으로 외부에서 온 메서드를 호출하기 때문에, 동기화 블록 안에서 사용하면 위험하다.
    - Stream을 사용해야 한다면, 복사본을 만들자

## 결론
- Lock을 소유한 상태에서는 내가 통제할 수 있는 가장 짧고 확실한 코드만 실행하라.
- 동기화 블록 안에서 외계인 메서드를 호출하지 마라. 꼭 필요하면 스냅샷을 복사해서 블록 밖에서 호출하라.
- 직접 Lock을 제어하기보다 `java.util.concurrent` 패키지의 유틸리티를 먼저 고려하라.
- 클래스를 설계할 때, 내부 동기화 vs 외부 동기화를 의식적으로 선택하라.
