# 추상화 수준에 맞는 예외를 던지라

메서드가 저수준 예외를 처리하지 않고 바깥으로 전파해버리는 경우 사용자들에게 혼란을 줄 수 있다.

예를 들어 `Service` 계층에서 `SQLException`이 발생한다면, 사용자들은 데이터베이스와 관련된 문제가 있다고 생각할 것이다.
하지만 이러한 예외를 사용자들은 알 필요도 없고 캡슐화를 위해서는 알아서는 안된다.

이 문제를 피하기 위해선 상위 계층에서는 저수준 예외를 잡아 자신의 추상화 수준에 맞는 예외로 바꿔 던져야 한다.
이를 `예외 번역` 이라 한다.

```java
try{
	...
	}catch(SQLException e) {
	    throw new ServiceException("데이터베이스 오류", e);
}
```
위 코드는 `SQLException`이 발생했을 때 `ServiceException`으로 번역해서 던지는 예시이다.
`ServiceException`은 `SQLException`을 감싸고 있기 때문에, 사용자들은 서비스 레이어에서 문제가 있다는 것을 알 수 있지만, `SQLException`이 발생했다는 것을 알 수는 없다.

### 예외 연쇄
예외 번역을 할 때, 원래 예외를 새 예외의 원인으로 포함시키는 것을 `예외 연쇄` 라고 한다.
저수준 예외가 디버깅에 도움이 되는 경우 예외 연쇄를 사용하는게 좋다.

```java
try{
	...
	}catch(SQLException ex) {
	    throw new ServiceException(ex);
}
```

위와 같은 방법을 통해 저수준 예외의 원인을 고수준 예외로 전달할 수 있고
필요 시 `getCause()` 메서드를 통해 원래 예외에 접근할 수 있다.

- `getCause()` : Throwable 클래스의 메서드로, 예외의 원인을 반환한다.

예외 연쇄를 사용하기 위한 고수준 예외의 생성자 정의
```java
public class ServiceException extends RuntimeException {
    public ServiceException(Throwable cause) {
        super(cause);
    }
}
```

대부분의 표준 예외는 예외 연쇄용 생성자를 갖추고 있다.
만약 그렇지 않더라도 `Throwable` 클래스의 `initCause()` 메서드를 사용하여 예외 연쇄를 구현할 수 있다.


** 예외를 전파하는 것보다 예외 번역이 우수하지만, 그렇다고 남용해서는 안된다.**

가장 좋은 방법은 최대한 저수준 메서드가 반드시 성공하도록하여 아래 계층에서 예외가 발생하지 않는 것이 최선이다.
이를 위해 상위 계층에서 아래 계층을 호출하기전 유효성 검사를 진행하는 방식을 활용할 수 있다.

### 내 생각
"이를 위해 상위 계층에서 아래 계층을 호출하기전 유효성 검사를 진행하는 방식을 활용할 수 있다." 라고 책에선 말하지만

고수준에서 유효성 검사를 진행하는 경우 여러 계층에서 동일한 유효성 검사를 반복해야 하는 문제가 발생할 수 있다.
뭐 이런 유효성 검사를 공통으로 처리하는 유틸리티 클래스를 만들어서 사용하면 되긴 하지만, 이 경우에도 유효성 검사 로직이 여러 곳에 분산되어 유지보수가 어려워질 수 있다고 생각한다.

그래서 뭐가 맞는 선택인지는 모르지만 이런 유효성 검사 등은 저수준 클래스의 책임이라고 생각한다.
때문에 정책이 변경되거나 관련 비즈니스 로직이 변경되었을 때 해당 저수준 클래스만 변경한다면 유지보수 측면에서 더 좋을 것 같다는 생각이 든다.

결론 상황에 맞게 선택하는게 좋겠지만, 개인적으로는 저수준 클래스에서 유효성 검사를 하는 편이 더 낫다고 생각한다.
PS. 어떠한 유효성 검사냐에 따라 Fail Fast 방식이 더 적합할 수도 있고, 예외 번역이 더 적합할 수도 있을 것 같다.
