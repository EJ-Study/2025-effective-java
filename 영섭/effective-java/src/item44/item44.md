# 표준 함수형 인터페이스를 사용하라

## 1. 표준 함수형 인터페이스를 권장하는 이유

* **API 일관성:** 개발자들이 익숙한 형식을 사용하여 API 학습 노출을 줄여줍니다.
* **상호운용성:** 다른 표준 API들과 자연스럽게 통합됩니다.
* **유지보수성:** 불필요한 코드(커스텀 인터페이스 정의)를 줄일 수 있습니다.

## 2. 반드시 알아야 할 기본 인터페이스 (6가지)

대부분의 상황은 이 6가지 기본 인터페이스로 해결됩니다.

| 인터페이스 | 메서드 시그니처 | 예시 | 설명 |
| --- | --- | --- | --- |
| **`UnaryOperator<T>`** | `T apply(T t)` | `String::toLowerCase` | 인수가 1개이며, 반환 타입이 인수와 같음 |
| **`BinaryOperator<T>`** | `T apply(T t1, T t2)` | `BigInteger::add` | 인수가 2개이며, 반환 타입이 인수와 같음 |
| **`Predicate<T>`** | `boolean test(T t)` | `Collection::isEmpty` | 인수를 받아 `boolean`을 반환 |
| **`Function<T, R>`** | `R apply(T t)` | `Arrays::asList` | 인수의 타입과 반환 타입이 다름 |
| **`Supplier<T>`** | `T get()` | `Instant::now` | 인수는 없고 값을 생성해서 반환 |
| **`Consumer<T>`** | `void accept(T t)` | `System.out.println` | 인수를 받고 소비만 함 (반환 없음) |

## 3. 커스텀 함수형 인터페이스는 언제 만들까?

1. **자주 쓰이며 이름 자체가 용도를 명확히 설명할 때:** (예: `Comparator`는 `BiFunction<T, T, Integer>`와 구조가 같지만 이름이 훨씬 직관적입니다.)
2. **반드시 지켜야 할 규약(Contract)이 있을 때:** 인터페이스가 따라야 하는 특별한 논리가 필요할 때입니다.
3. **전용 디폴트 메서드를 제공해야 할 때:** (예: `Comparator`의 `reversed()`, `thenComparing()` 등)

## 💡 핵심 주의사항

* **기본 타입용 인터페이스 활용:** `Function<Integer, Integer>`처럼 박싱된 타입을 쓰기보다 `IntFunction`, `LongBinaryOperator` 같은 **기본 타입 전용** 인터페이스를 사용하세요. 성능 저하(오토박싱)를 피할 수 있습니다.
* **`@FunctionalInterface` 애너테이션:** 직접 인터페이스를 만든다면 반드시 이 애너테이션을 붙이세요. 컴파일 시점에 함수형 인터페이스인지 체크해주고, 나중에 누군가 메서드를 추가하는 실수를 막아줍니다.
* **다중정의(Overloading) 회피:** 서로 다른 함수형 인터페이스를 인수로 받는 메서드들을 다중정의하지 마세요. 클라이언트 코드에서 모호함이 발생해 컴파일 오류나 혼란을 야기할 수 있습니다.
