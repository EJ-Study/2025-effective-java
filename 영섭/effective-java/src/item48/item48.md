# 스트림 병렬화는 주의해서 적용하라

## 1. 병렬화로 효과를 보기 어려운 경우

* **데이터 소스가 `Stream.iterate`이거나 중간 연산에 `limit`을 쓸 때:**
  데이터를 어떻게 나눌지 모르는 상태에서 병렬화를 시도하면 시스템은 데이터를 순차적으로 탐색하며 나누려 시도하느라 CPU만 낭비하게 됩니다.
* **종단 연산의 작업량이 많을 때:**
  병렬화의 이점은 작업을 나누는 비용보다 작업을 처리하는 이득이 클 때 발생합니다. 처리량이 적으면 오히려 나누는 비용이 더 큽니다.

## 2. 병렬화가 효율적인 경우

* **데이터 소스가 정확히 분할될 때:**
  `ArrayList`, `HashMap`, `HashSet`, `ConcurrentHashMap`의 인스턴스거나 배열, int 범위, long 범위일 때 효과가 좋습니다. 이들은 데이터 크기를 정확히 알 수 있고, 원하는 위치로 바로 접근이 가능하기 때문입니다.
* **참조 지역성(Locality of Reference)이 높을 때:**
  데이터들이 메모리에 연속해서 저장되어 있으면 벌크 연산 시 성능이 극대화됩니다. 기본 타입 배열이 가장 우수합니다.
* **종단 연산이 가벼울 때:**
  축약(Reduction) 연산인 `min`, `max`, `count`, `sum`이나 조건에 맞는지 확인하는 `anyMatch`, `allMatch` 등은 병렬화에 적합합니다.

## 3. 병렬화 시 주의사항 (안전성)

병렬화를 적용할 때는 반드시 **결과의 정확성**을 확인해야 합니다.

* **Safety Failure:** 병렬화된 스트림에서 상태를 공유하는 함수를 사용하거나, 스트림의 규약을 어기면 결과가 틀리거나 예상치 못한 동작을 할 수 있습니다.
* **성능 측정 필수:** 병렬화는 "성능 최적화"의 수단일 뿐입니다. 적용 전후를 반드시 **JMH(Java Microbenchmark Harness)** 같은 도구로 측정하여 실제로 빨라졌는지 확인해야 합니다.

> **결론:** 병렬 스트림은 성능을 드라마틱하게 높여줄 수 있는 도구지만, 잘못 쓰면 버그를 만들고 성능을 갉아먹습니다. **확신이 없다면 병렬화를 시도하지 않는 것이 상책입니다.**
