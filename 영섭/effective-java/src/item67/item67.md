## 아이템 67. 최적화는 신중히 하라

### 핵심 메시지

최적화는 좋은 결과보다 **해로운 결과**를 낳는 경우가 많다. 특히 빠르지도 않고 올바르지도 않으며 수정하기도 어려운 소프트웨어가 탄생한다.

---

### 유명한 격언 3가지

> "맹목적인 어리석음을 포함한 그 어떤 이유보다 **효율성**이라는 이름 아래 더 많은 죄악이 저질러진다."
> — 윌리엄 울프

> "작은 효율성은 모두 잊어라. 섣부른 최적화가 만악의 근원이다."
> — 도널드 크누스

> "최적화를 할 때는 두 가지 규칙을 따르라. 첫 번째, 하지 마라. 두 번째, 아직 하지 마라."
> — M. A. 잭슨

---

### 성능보다 좋은 아키텍처가 우선이다

성능을 위해 **아키텍처 원칙을 희생하지 마라.** 빠른 프로그램보다 좋은 프로그램을 작성하라.

좋은 프로그램은 **정보 은닉** 원칙을 따르기 때문에, 나중에 각 요소를 독립적으로 최적화할 수 있다.

---

### 설계 단계에서 성능을 고려해야 할 부분

나중에 바꾸기 어려운 설계 결정들은 처음부터 신중하게 해야 한다.

**1. API 설계**
```java
// ❌ 나쁜 예 - 구체 타입 반환 (나중에 바꾸기 어려움)
public LinkedHashSet<String> getItems() { ... }

// ✅ 좋은 예 - 인터페이스 반환 (유연하게 최적화 가능)
public Set<String> getItems() { ... }
```

**2. 가변 타입 노출 금지**
```java
// ❌ 나쁜 예 - 내부 가변 객체를 직접 노출
public Date getDate() {
    return this.date; // 외부에서 수정 가능
}

// ✅ 좋은 예 - 방어적 복사
public Date getDate() {
    return new Date(this.date.getTime());
}
```

**3. 네트워크 프로토콜, DB 스키마** 등도 한번 공개되면 바꾸기 매우 어렵다.

---

### 최적화 실행 순서

```
1. 좋은 설계로 프로그램 작성
        ↓
2. 완성 후 성능 측정 (프로파일링)
        ↓
3. 병목 지점 파악
        ↓
4. 최적화 시도
        ↓
5. 다시 성능 측정 (개선됐는지 반드시 확인)
```

> 최적화 시도 전후로 **반드시 성능을 측정하라.** 시도한 최적화가 실제로 효과가 있었는지 확인해야 한다.

---

### 자바는 성능 예측이 특히 어렵다

자바는 기본 연산의 성능이 **구현에 따라 크게 다르다.**

- JVM마다 성능이 다르다
- JIT 컴파일러 최적화가 동적으로 이루어진다
- 같은 코드도 실행 환경에 따라 성능 차이가 발생한다

따라서 **프로파일링 도구**를 활용해 실제 병목 지점을 찾아야 한다.

---

### 정리

| 단계 | 행동 |
|------|------|
| 설계 시 | API, 프로토콜, 스키마 등 변경 어려운 부분만 성능 고려 |
| 구현 시 | 좋은 설계와 원칙에 집중, 최적화 금지 |
| 완성 후 | 프로파일러로 병목 지점 측정 후 최적화 |
| 최적화 후 | 반드시 재측정해서 효과 확인 |

> **핵심 한 줄 요약:** 완성된 프로그램을 측정하고, 실제 병목에만, 신중하게 최적화하라.
